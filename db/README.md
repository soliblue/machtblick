# Database

All apps and ETL workers share one SQLite database, `db/machtblick.sqlite` (gitignored), opened through `db/client.ts`. Set `MACHTBLICK_DB` to point at a different file. The Drizzle schema lives in `db/schema/`, migrations in `db/migrations/`.

The migration journal is currently incomplete, so `npm run db:migrate` does not guarantee a full cold-clone schema. Until the journal and snapshots are reconciled, inspect generated SQL and journal changes before applying a migration.

## Getting a working database

A cold clone has no complete database bootstrap path. Request a database snapshot from the operator at feedback@machtblick.de rather than relying on `npm run db:migrate` alone.

Regenerating the production dataset from scratch is not a supported contributor path. It requires:

- Source imports against Bundestag Open Data, DIP, abgeordnetenwatch.de, and Wikidata (hours of fetching, plus one-shot historical imports under `etl/_oneshot/` via the `oneshot:*` scripts)
- Derived fields (simplified titles, summaries, classifications, party positions, English translations) generated by local LLM agent CLIs (`codex exec`, `claude -p`) over thousands of documents, days of runtime
- The exact incremental ordering documented in `prompts/auto-refresh.md` (normalization steps are order-sensitive)

Place the operator snapshot at `db/machtblick.sqlite`. With the snapshot in place, `npm run dev -w @machtblick/bundestag` works directly, and `npm run build -w @machtblick/bundestag` regenerates member photos, OG images, and static JSON from it.

## Refreshing and normalizing

Incremental refreshes follow `prompts/auto-refresh.md`, driven weekly by a systemd timer (`scripts/systemd/`), operator-run. The `db:*` npm scripts in the root `package.json` are idempotent, checked-in normalization and materialization steps; every data correction must be reproducible through one of them or an importer, never a manual edit.
