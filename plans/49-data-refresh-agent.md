# Data refresh agent

Status: Draft.

Goal: Define a recurring `refresher` specialist that can be launched manually every couple of weeks to refresh Bundestag data, run the generated summary and translation jobs, audit coverage, and hand lead a short release-ready report.

This plan is for the agent and runbook design only. Implementing the agent, adding orchestration scripts, and changing ETL behavior should happen as follow-up work under this plan or a smaller implementation plan.

## Shared contracts

- `.claude/agents/refresher.md` is the source instruction file for the new specialist.
- `.codex/agents/refresher.toml` is generated by `npm run agents:sync`.
- `refresher` owns orchestration, refresh reporting, and first-pass audits.
- `plumber` still owns schema, migrations, source-specific ETL workers, and data-shape fixes.
- `backend` and `frontend` own app contract or UI changes if refreshed data exposes a missing surface.
- `tester`, `visibility`, and `deployer` remain separate gates. `refresher` does not deploy.
- Refreshed data lives in `db/machtblick.sqlite` and generated build output. The SQLite DB and generated public JSON are ignored artifacts, not git artifacts.
- Every LLM data job uses local agent CLIs through the existing ETL scripts, preferably `codex exec`.
- Read paths keep trusting normalized DB data. If refreshed data needs compensation, the fix belongs in ETL or a DB normalization script.

## Proposed agent responsibility

`refresher` should be a data operations agent, not a second lead.

It should:

- Read this plan, `AGENTS.md`, `.claude/agents/plumber.md`, and the current ETL scripts before the first run.
- Start each refresh with a dated log entry in this plan or a dated run plan if the refresh is expected to lead to deployable data changes.
- Run deterministic data pulls in a fixed order.
- Run generated summary, title, position, and translation jobs after source data is current.
- Record counts before and after the refresh.
- Detect anomalies, missing coverage, and upstream shape changes.
- Make tiny runbook fixes only when the cause is obvious and scoped.
- Open a follow-up plan for schema, ETL, backend, frontend, or design work when the issue is bigger than a refresh run.
- Produce a final report with commands run, counts changed, skipped rows, failures, and recommended next action.

It should not:

- Deploy.
- Change app UI during a routine refresh.
- Add new schema without handing the work to plumber.
- Patch read paths to hide data quality issues.
- Re-run expensive LLM jobs with `--force` unless lead explicitly asks.

## Refresh sequence

### Preflight

- Confirm the worktree state and note unrelated local changes.
- Confirm `db/machtblick.sqlite` exists and is the intended target.
- Capture baseline counts:
  - votes by `vote_type`, newest vote date, and newest `bundestag_id`
  - speeches by newest date, sessions covered, unmatched speakers
  - Anfragen count by type and newest `updated_at`
  - Antraege count by type, newest `updated_at`, count missing descriptions
  - `vote_antraege` count and distinct linked vote count
  - translation coverage for votes, party summaries, speeches, and Antraege
- Decide whether to run DIP full-cache mode or incremental mode with `DIP_UPDATED_START`.
- Create a local DB copy before writes if the refresh is more than a small incremental run.

### Source data refresh

Run the source workers in dependency order:

1. `npm run etl:stammdaten`
2. `npm run etl:abgeordnetenwatch`
3. `npm run etl:votes:namentlich`
4. `npm run etl:handzeichen:refresh`
5. `DIP_UPDATED_START=<last-refresh-date> npm run etl:dip` when incremental is enough, otherwise `npm run etl:dip`
6. `npm run etl:dip:answers`
7. `npm run etl:speeches:xml`
8. `npm run etl:affiliations`
9. `npm run db:normalize`

Less frequent or conditional jobs:

- `npm run etl:donations` when a new party donation year or announcement is expected.
- `npm run etl:portraits` when members changed or portraits are missing.
- `npm run etl:terms` when Bundestag term metadata changes.

### Derived data refresh

Run derived jobs after all source rows and links are current:

1. `npm run etl:titles`
2. `npm run etl:antrag-titles`
3. `npm run etl:descriptions`
4. `npm run etl:antrag-descriptions`
5. `npm run etl:party-positions`
6. `npm run etl:translations`
7. `npm run etl:antrag-description-translations`
8. `npm run etl:speech-translations`

Use default stale detection. Do not pass `--force` during routine refreshes.

Current gap: the repo has full speech text, speech translations, and party-position summaries derived from speeches. It does not yet have a dedicated per-speech summary table. If "speech summaries" means summaries of individual speeches, plumber should first add a schema and ETL contract for that before refresher includes it in the routine.

## Required audits

### Vote audits

- New namentlich, handzeichen, and hammelsprung vote counts are plausible for the date window.
- `votes.result` is normalized after refresh.
- `votes.procedural` was applied to newly ingested procedural titles.
- `audit-self-no.mjs` and `audit-suspicious-initiator.mjs` are clean after handzeichen refresh.
- New handzeichen rows have proposer enrichment when a Drucksache is present.
- New non-procedural votes have `clean_title`, summaries, translations, and JSON endpoint output when eligible.

### Speech audits

- XML sessions fetched up to the newest available session, stopping only after the expected consecutive 404s.
- `agenda_item` coverage is reported.
- `vote_id` linkage coverage is reported.
- Top unmatched non-role speakers are reviewed.
- New speeches have English translations when eligible.
- Party-position summary jobs ran for new namentlich votes with enough party speech text.

### Antrag and Anfrage audits

- `antraege.beratungsstand` changed for pending items when DIP changed status.
- Previously non-final Antraege that now have a vote are linked through `vote_antraege`.
- Member detail motion status follows DB data, not route-specific status logic.
- Newly introduced Antraege have signatories where DIP or PDF blocks expose them.
- Newly introduced Antraege have generated descriptions and English translations when PDFs are available.
- Anfragen answers were refreshed and extracted.

If the product needs "what changed since the last refresh" on member pages, add a small refresh snapshot table or generated report first. The current schema stores current status, not a status history.

### Member audits

- New MdBs, replacement mandates, and ended mandates are reflected in `members`, `member_mandates`, and `member_affiliations`.
- Open affiliations are still plausible after the newest vote date.
- Long gaps in roll-call participation are reported, not silently treated as mandate ends unless source data confirms it.
- Member pages for newly seen members are included in prerender paths after build.

### Build audits

- Run `npm run build -w @machtblick/bundestag`.
- Confirm prerender includes new vote, motion, member, party, and English paths.
- Confirm generated static JSON exists for new votes, motions, members, parties, and speech search shards.
- If user-visible behavior changed because data shape changed, run tester before any deploy request.
- If deploy is requested later, run visibility before deployer.

```md
- ...

- votes: before N, after N, newest YYYY-MM-DD
- speeches: before N, after N, newest YYYY-MM-DD

Generated jobs:
- descriptions: completed N, skipped N, failed N
- party positions: completed N

- suspicious initiator: pass

- ...
```

## Automation options

### Option 1: manual agent launch

Lowest risk.

Lead manually launches `refresher` every couple of weeks with a prompt that points to this plan. The agent runs the full sequence, appends a report, and stops before deploy. This is the best first version because the refresh includes network calls, DB writes, LLM jobs, and possible upstream surprises.

### Option 2: deterministic refresh script plus manual agent review

Good next step.

Add a single script, likely `scripts/refresh-bundestag-data.mjs`, that runs the deterministic commands, captures logs under an ignored directory, uses a lockfile, and exits non-zero on any failed audit. Schedule that script with cron or a systemd timer on the machine that already has the DB, caches, Codex credentials, and deploy tooling.

Then launch `refresher` manually to inspect the log, run derived LLM jobs if needed, and decide whether the result is deployable.

### Option 3: scheduled local Codex run

Possible, but only after the manual runbook is stable.

A cron or systemd timer can run something like:

```sh
codex exec --cd . --sandbox danger-full-access -a never "Read plans/49-data-refresh-agent.md and run the recurring Bundestag data refresh. Stop before deploy and write a report."
```

Requirements:

- the runner has the same repo checkout and persistent `db/machtblick.sqlite`
- Codex CLI is logged in and available non-interactively
- network access to Bundestag, DIP, abgeordnetenwatch, Wikidata, and any model CLI endpoints
- a lockfile prevents overlapping runs
- logs are retained outside gitignored generated output
- notifications go somewhere visible when the run fails
- deploy remains a separate explicit human action

### Option 4: GitHub Actions schedule

Useful only for deterministic checks unless a self-hosted runner is used.

Hosted GitHub runners do not have the persistent SQLite DB, ETL caches, local Codex login, or long-running LLM context by default. A hosted cron could run lightweight audits or build from committed source, but a real refresh would need either:

- a self-hosted runner on the data machine, or
- durable artifact storage for the DB and caches, plus secure non-interactive model credentials.

For this project, a self-hosted runner or systemd timer is simpler than hosted Actions.

### Option 5: external scheduler trigger

Cloudflare Cron, GitHub schedule, or another hosted scheduler can trigger a webhook on the data machine. The webhook should only enqueue the local refresh script. It should not run ETL inside Cloudflare, because the jobs are too long, need filesystem state, and write the local SQLite database.

## Implementation work

1. Add `.claude/agents/refresher.md`.
2. Run `npm run agents:sync`.
3. Update `AGENTS.md` and `CLAUDE.md` specialist tables.
4. Add `scripts/refresh-bundestag-data.mjs` once the manual command order has been verified.
5. Add `scripts/audit-refresh.mjs` for reusable count and coverage checks.
6. Add a `package.json` script such as `refresh:bundestag`.
7. Decide where ignored run logs live, for example `.cache/refresh-runs/`.
8. Decide whether each real refresh appends to this plan or creates a dated run plan.
9. After two clean manual runs, add a systemd timer or cron entry on the data machine.

## Open questions

- Should the specialist be named `refresher`, `data-refresh`, or `operator`?
- Should routine refreshes create dated plan files, or is appending to this plan enough?
- Do we want dedicated per-speech summaries, or are party-position summaries the intended speech-derived summaries?
- What is the desired notification channel for failed scheduled refreshes?
- Should deploy ever be offered at the end of a clean refresh, or should deploy stay a separate user request only?
