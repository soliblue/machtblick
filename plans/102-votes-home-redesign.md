## Phase 2: divergent directions (2026-07-04)
User verdict after 5 rounds: "still mediocre". Incremental critique converged on safe. New approach: 3 independent designers each build a self-contained HTML prototype of /votes/ in a radically different direction. User picks; winner gets implemented in React.
- Direction A "Zeitung": editorial front page. Newspaper typography, headline hierarchy, the day's votes as news. Think FAZ/Zeit election coverage.
- Direction B "Ticker": dense data ledger. Monospace numerals, tight rows, everything scannable, terminal/Bloomberg energy but within the design tokens.
- Direction C "Plakat": bold poster typography. Huge numbers, the result IS the design, one vote = one strong visual statement, scroll rhythm of a feed.
Prototypes: /tmp/votes-directions/{a,b,c}.html + screenshots /tmp/votes-directions/{a,b,c}-{mobile,desktop}.png. Constraints for all: mobile-first 390px + desktop adaptation, design tokens from CLAUDE.md, no left-edge indicators, color = meaning, real data copied from the live page at :5174/votes/.

## Phase 4: unification (2026-07-04)
User hates having two separate designs (mobile card language vs desktop ledger language). Decision: ONE ingredient set, the card's (stamp, hemicycle, donut row, markdown summary, proposer kicker). Desktop options to explore, per user: (A) centered feed column of the same card, but desktop viewports are short/landscape so cards must be natural-height, not viewport-height; (B) wide card rows, same ingredients stacked horizontally instead of vertically. Prototype BOTH, user picks. The agate table / stat trio / dek ledger language dies either way. Amendments 20+ since the tester pass: stamp straight (rotated=false on the card), stamp inline in kicker row (right-aligned, no overlay), summary directly below title, party donuts row (PartyDonutRow, sorted yes-left to no-right, size 52) replacing mini bars, summary clip 700 chars, fitted line clamp with instant pre-hydration ellipsis via head stylesheet.

RESOLVED phase 4 (2026-07-04): user picks B with a modified split: desktop card = two columns, LEFT everything up to and including the summary (kicker with proposer+date+stamp, title, summary), RIGHT everything after the summary stacked vertically (hemicycle with flanking counts + legend, then donut row). Same card component family, natural height, daylines kept. Ledger components (LedgerRow, PartyAgateTable, stat trio, dek-as-dek) retire; deriveDescription stays as summary fallback only.

## Iteration protocol
Each round the designer critique must name what is boring/weak and change something structural, not just polish. Rounds should explore: information hierarchy (what is the ONE thing a citizen scans for), result visualization (stacked bar idiom), scannability rhythm, topic/party salience, visual interest without decoration (color = meaning).
